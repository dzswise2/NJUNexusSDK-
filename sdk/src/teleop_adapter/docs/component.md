# Table Arm Driver Node

## 功能概述
Table Arm Driver Node 是一个用于控制机械臂的 ROS2 节点，负责建立 ROS2 与机械臂硬件之间的通信桥梁。它的主要功能包括：
- 读取和解析 ROS2 启动参数（配置文件），验证配置的合理性。
- 根据配置参数初始化机械臂适配器。
- 通过适配器 API 处理 ROS2 的消息发布和订阅，实现上层控制指令下发和底层状态反馈。

## 输入输出接口
- **输入**：
  - **配置文件**：`table_arm.yaml` 或 `table_arm_single.yaml`（通过 ROS2 参数系统加载）。
  - **ROS2 订阅话题**：`rt/teleop/table_arm/joint_cmd`（机械臂关节控制命令）。
- **输出**：
  - **ROS2 发布话题**：
    - `rt/teleop/table_arm/joint_state`：机械臂的关节状态（位置、速度、力矩等）。
    - `rt/teleop/table_arm/status`：机械臂的运行状态和错误信息。

## 运行逻辑

### 1. 读取 ROS2 参数
   - 节点启动时，通过 ROS2 参数系统读取 `table_arm.yaml` 或 `table_arm_single.yaml` 配置文件。
   - 解析并验证配置文件中的参数，包括：
     - `num_of_dofs`：所有机械臂的自由度总数。
     - `default_dof_pos`：所有关节的默认初始位置（数组长度需与 `num_of_dofs` 一致）。
     - `joint_names`：所有关节的名称列表（数组长度需与 `num_of_dofs` 一致）。
     - `joint_mapping`：关节映射索引（数组长度需与 `num_of_dofs` 一致）。
     - `*_arm_*` 参数：单个机械臂的配置参数（如 `right_arm_uart_port`、`right_arm_num_of_dofs` 等）。
   - 执行配置合理性交叉验证：
     - 验证总配置（`num_of_dofs`、`joint_names` 等）与各单臂配置（`*_arm_num_of_dofs`）的一致性。
     - 检查数组长度是否匹配，关节名称是否重复，映射索引是否有效等。
   - **验证结果**：
     - 配置合理：继续执行后续步骤。
     - 配置不合理：打印详细错误信息，停止节点运行。

### 2. 根据参数初始化适配器
   - 识别配置文件中所有 `<arm_name>_arm_*` 格式的参数，确定需要初始化的机械臂数量：
     - 遍历配置参数，识别所有不同的 `<arm_name>` 前缀（如 `right_arm_*`、`left_arm_*`、`center_arm_*` 等）。
     - 每个唯一的 `<arm_name>` 前缀对应一个机械臂适配器。
     - 示例：
       - `table_arm_single.yaml`：只有 `right_arm_*` → 初始化 1 个适配器。
       - `table_arm.yaml`：有 `right_arm_*` 和 `left_arm_*` → 初始化 2 个适配器。
       - 支持扩展到 N 个机械臂（如 `right_arm_*`、`left_arm_*`、`center_arm_*`、`back_arm_*` 等）。
   - 为每个识别到的机械臂创建独立的适配器对象，每个适配器在初始化时执行以下操作：
     - 根据对应的 `<arm_name>_arm_uart_port` 参数打开串口设备。
     - 创建串口读写线程，建立与硬件设备的通信通道。
     - 根据 `<arm_name>_arm_num_of_dofs`、`<arm_name>_arm_feedback_rate` 等配置参数设置设备工作模式。
     - 完成设备自检和状态初始化。
   - **初始化结果**：
     - 初始化成功：所有 N 个适配器都进入就绪状态，可接受上层调用。
     - 初始化失败：打印详细错误信息（如串口打开失败、设备无响应、具体哪个机械臂初始化失败等），停止节点运行。

### 3. 调用适配器 API 处理 ROS2 发布订阅
   - 创建 ROS2 通信线程，通过适配器提供的统一 API 接口与硬件交互：
     - **发布线程**：
       - 定期调用适配器 API 读取机械臂的关节状态和设备状态。
       - 将读取的数据封装为 ROS2 消息并发布到相应话题：
         - `rt/teleop/table_arm/joint_state`：发布关节状态。
         - `rt/teleop/table_arm/status`：发布设备状态。
     - **订阅线程**：
       - 订阅 `rt/teleop/table_arm/joint_cmd` 话题接收控制命令。
       - 将接收到的命令通过适配器 API 下发给机械臂硬件。
   - 所有与硬件的交互均通过适配器的统一抽象接口完成，ROS2 节点层无需了解底层通信协议细节。

## 错误处理
- **配置文件解析失败**：打印详细的参数错误信息，停止节点运行。
- **适配器初始化失败**：打印设备初始化错误（如串口、通信超时等），停止节点运行。
- **运行时错误**：适配器内部处理通信异常，并通过状态话题上报错误信息。

## 适配器软件分层设计

机械臂适配器采用三层软件架构设计，实现了从底层硬件通信到上层统一接口的封装。从底层到上层依次为：串口通讯层、设备应用层和统一抽象层。

### 第一层：串口通讯层（基类）
- **职责**：
  - 负责底层串口的物理通讯，实现传输层的数据收发功能。
  - 管理串口设备的初始化、配置（波特率、数据位、停止位等）。
  - 提供线程安全的读写缓冲区，供上层调用。
  - 处理串口通讯中的错误检测和异常恢复。
- **特点**：
  - 作为基类存在，不包含具体设备的协议逻辑。
  - 只负责字节流的传输，不关心数据的含义。

### 第二层：设备应用层
- **职责**：
  - 继承串口通讯层，在其基础上实现具体设备的应用层通讯协议。
  - 负责数据包的组包（编码）和解包（解码），处理协议的校验、帧头帧尾等。
  - 解析设备返回的原始数据，转换为有意义的状态信息（如电机位置、速度、力矩等）。
  - 将上层的控制指令编码为设备能够理解的通讯协议格式。
  - 提供设备特定的操作接口，包括：
    - 电机通讯接口：读取电机状态、发送控制命令。
    - 设备配置接口：设置设备参数、工作模式等。
    - 设备状态获取接口：查询设备运行状态、错误信息。
    - 传感器数据获取接口：读取力传感器、IMU 等传感器数据。
- **特点**：
  - 与具体硬件设备强相关，不同设备需要实现不同的应用层类。
  - 通过调用串口通讯层的接口完成实际的数据传输。

### 第三层：统一抽象层
- **职责**：
  - 定义统一的设备操作抽象接口，屏蔽不同设备的实现差异。
  - 为上层（ROS2 节点、初始化逻辑等）提供一致的调用方式。
  - 通过多态机制，由设备应用层的具体类实现抽象接口。
- **提供的抽象能力**：
  - 电机通讯抽象：统一的电机状态读取和命令发送接口。
  - 设备操作抽象：统一的设备配置、状态查询接口。
  - 传感器数据抽象：统一的传感器数据获取接口。
- **特点**：
  - 与具体设备无关，只定义接口规范。
  - 上层调用者无需了解底层设备的通讯协议细节。

### 适配器内部的层次关系与协作流程
- **依赖关系**：
  - 设备应用层依赖串口通讯层，继承其基类并调用其传输接口。
  - 统一抽象层定义接口规范，由设备应用层具体实现。
  - ROS2 节点层只依赖适配器的统一抽象层，通过抽象接口操作设备，无需关心底层实现。

- **适配器内部数据流向**：
  - **下行（命令发送）**：统一抽象层接口 → 设备应用层（组包）→ 串口通讯层（发送）→ 硬件设备
  - **上行（状态读取）**：硬件设备 → 串口通讯层（接收）→ 设备应用层（解包）→ 统一抽象层接口

- **适配器初始化流程**：
  1. 串口通讯层完成串口的打开和参数配置，启动读写线程。
  2. 设备应用层加载设备特定配置，初始化通讯协议状态机。
  3. 统一抽象层对外暴露接口，供 ROS2 节点层调用。

- **适配器运行时协作**：
  1. ROS2 节点层通过适配器统一抽象层的接口读取设备状态。
  2. ROS2 节点层通过适配器统一抽象层的接口下发控制命令。
  3. 适配器内部：设备应用层根据协议要求，通过串口通讯层与硬件设备交互。

### 适配器分层设计的优势
通过这种适配器内部分层设计，实现了：
- **高内聚**：适配器内每层职责明确，只关注自己的功能实现。
- **低耦合**：层与层之间通过接口交互，易于替换和扩展。
- **可维护性**：新增不同型号的机械臂设备时，只需实现新的设备应用层，无需修改串口通讯层和上层 ROS2 节点代码。
- **可测试性**：适配器的每层可以独立进行单元测试和集成测试。
- **设备无关性**：ROS2 节点层通过统一抽象接口操作设备，不依赖具体硬件型号和通信协议。

## 实现细节

### 第一层：通讯层

#### 通讯接口抽象（CommunicationInterface）

定义统一的通讯接口，支持多种硬件传输层：串口、以太网、CAN等。

**核心接口**：
- `open()` / `close()`：打开/关闭通讯端口
- `read()` / `write()`：读写数据
- `isOpen()`：检查连接状态
- `getBytesAvailable()`：获取可读字节数
- `clearBuffer()`：清空缓冲区
- `getLastError()`：获取错误信息
- `getType()`：获取通讯类型标识

**已实现文件**：
- `include/teleop_adapter/communication_interface.hpp`

#### 串口通讯实现（SerialPort）

实现 `CommunicationInterface` 接口，提供线程安全的串口读写能力。

**核心功能**：
- 串口管理：打开/关闭串口设备，配置波特率、数据位、停止位、校验位等参数
- 数据传输：非阻塞读取数据，线程安全写入数据，支持超时控制
- 状态查询：检查串口连接状态，获取可读字节数，错误状态报告

**关键实现特性**：
- 使用非阻塞IO模式（`O_NONBLOCK`）避免读取阻塞
- 使用 `select` 机制实现超时读取
- 写入操作一次性完成，不重试，失败立即报错
- 使用互斥锁保护并发访问，支持多线程环境

**已实现文件**：
- `include/teleop_adapter/serial_port.hpp`
- `src/serial_port.cpp`

**扩展性**：未来可以实现其他通讯方式，如：
- `EthernetPort`：以太网通讯（TCP/UDP）
- `CanPort`：CAN总线通讯
- `UsbPort`：USB通讯

### 第二层+第三层：统一设备适配器（DeviceAdapter）

采用工厂模式设计，将设备应用层（通讯协议）和统一抽象层（用户接口）合并实现，形成一个完整的设备适配器基类。

#### 设计思路
不再严格区分"设备应用层"和"统一抽象层"，而是设计一个统一的设备适配器基类，直接提供：
1. **硬件通讯接口**：基于 `CommunicationInterface` 抽象，支持串口、以太网、CAN等
2. **通讯协议实现**：数据包的编码解码
3. **电机控制接口**：电机命令发送、状态读取
4. **传感器接口**：力传感器、IMU等数据获取
5. **设备管理接口**：设备配置、状态查询、错误处理

#### 核心接口设计
设备适配器工厂类提供统一的抽象接口，具体设备（如Table Arm）继承此类并实现具体协议：

**连接管理**
- 初始化设备连接
- 打开/关闭串口
- 检查连接状态
- 设备握手和参数配置

**电机控制**
- 使能/失能电机
- 发送电机控制命令（位置、速度、力矩、刚度、阻尼）
- 读取电机状态（位置、速度、力矩）
- 设置电机参数（反馈频率、控制模式等）

**传感器数据**
- 读取力传感器数据
- 读取IMU数据
- 读取其他扩展传感器

**设备状态**
- 获取设备连接状态
- 获取设备错误信息
- 获取设备健康状态
- 重置设备错误

**线程管理**
- 内部管理读写线程
- 线程安全的数据缓冲
- 自动处理数据收发

#### 实现优势
1. **接口统一**：上层（ROS2节点）只需面对统一的接口，无需关心底层协议差异
2. **扩展方便**：新增设备类型只需继承基类，实现具体协议方法
3. **职责清晰**：设备适配器负责所有与硬件相关的操作，节点层只负责ROS通信
4. **易于测试**：可以创建Mock适配器用于无硬件测试

#### 已实现的基类功能

**DeviceAdapter 基类**提供以下功能模块：

1. **数据结构定义**
   - `DeviceAdapterConfig`：设备配置参数（设备名称、串口、波特率、自由度、反馈频率等）
   - `MotorState`：电机状态（位置、速度、力矩、使能状态、错误码）
   - `MotorCommand`：电机控制命令（MIT模式：kp、position、kd、velocity、torque）
   - `SensorData`：传感器数据（力传感器、IMU）
   - `DeviceStatus`：设备状态（连接、初始化、电机使能、错误信息）

2. **连接管理**（已实现）
   - `initialize()`：初始化适配器，打开串口，启动读写线程
   - `shutdown()`：关闭适配器，停止线程，关闭串口
   - `isConnected()`：检查连接状态

3. **电机控制接口**（纯虚函数，需具体设备实现）
   - `enableMotors()`：使能所有电机
   - `disableMotors()`：失能所有电机
   - `sendMotorCommands()`：发送电机控制命令
   - `readMotorStates()`：读取所有电机状态

4. **传感器接口**（虚函数，可选实现）
   - `readSensorData()`：读取传感器数据，默认返回不支持

5. **设备管理接口**（已实现）
   - `getDeviceStatus()`：获取设备状态
   - `getConfig()`：获取配置参数
   - `clearErrors()`：清除设备错误
   - `getNumDofs()`：获取自由度数量
   - `getArmName()`：获取机械臂名称

6. **协议相关接口**（纯虚函数，需具体设备实现）
   - `deviceHandshake()`：设备握手和初始化配置
   - `configureDevice()`：配置设备参数
   - `readThreadFunc()`：读线程处理函数
   - `writeThreadFunc()`：写线程处理函数

#### 使用方式

**创建带串口的设备适配器**：
```cpp
// 1. 创建并配置串口通讯对象
auto serial_port = std::make_unique<SerialPort>();
SerialPort::Config serial_config("/dev/ttyUSB0", 115200);
serial_port->open(serial_config);  // 先配置并打开串口

// 2. 创建设备配置
DeviceAdapterConfig device_config;
device_config.device_name = "right_arm";
device_config.num_of_dofs = 6;
device_config.feedback_rate = 400;
device_config.auto_enable = true;

// 3. 创建设备适配器，转移通讯对象的所有权
auto adapter = std::make_unique<TableArmAdapter>(
    device_config,
    std::move(serial_port)  // 所有权转移
);

// 4. 初始化适配器（如果串口未打开，会自动打开）
adapter->initialize();
```

**创建带以太网的设备适配器（未来）**：
```cpp
// 使用以太网通讯
auto ethernet = std::make_unique<EthernetPort>();
EthernetPort::Config eth_config("192.168.1.100", 8080);
ethernet->open(eth_config);

auto adapter = std::make_unique<RobotAdapter>(
    device_config,
    std::move(ethernet)
);
```

**关键点**：
- 通讯接口在传入 `DeviceAdapter` 前需要先配置好参数
- 使用 `std::move` 转移所有权，避免资源重复管理
- `DeviceAdapter` 不关心具体是什么通讯方式，只使用抽象接口

### 消息定义

#### sensor_msgs/JointState (ROS2 标准消息)
直接使用 ROS2 标准的 `sensor_msgs/msg/JointState` 消息类型，包含：
- `std_msgs/Header header`
- `string[] name` - 关节名称
- `float64[] position` - 关节位置
- `float64[] velocity` - 关节速度
- `float64[] effort` - 关节力矩

#### teleop_msg/ArmStatus (自定义状态消息)
用于上报机械臂运行状态和错误信息：
```
std_msgs/Header header
string[] arm_names      # 机械臂名称列表
bool[] connected        # 各机械臂连接状态
string[] error_msgs     # 错误描述信息
int32[] error_codes     # 错误代码
```

### 配置参数说明

#### control_type 参数
- `closed_loop`: 机械臂有板载闭环控制器（重力补偿、力反馈等）
- `open_loop`: 机械臂无控制器暴露，适配器直接读取关节状态
- **注意**: 此参数不在 table-arm-driver-node 节点中处理，供其他上层节点使用

#### feedback_rate 参数
- 含义：机械臂硬件的数据反馈频率（Hz）
- 用途：在适配器初始化时配置给机械臂设备
- 示例：`feedback_rate: 400` 表示 400Hz 的状态反馈

### Table Arm 机械臂串口通讯协议

#### 协议概述
Table Arm 机械臂采用基于串口的专用通讯协议，实现主机与机械臂控制器（MCU）之间的实时数据交换。

#### 物理接口规范
- **接口形式**: 3.3V TTL电平 UART
- **波特率**: 4Mbps (4,000,000 bps)
- **数据位**: 8bit
- **停止位**: 1bit  
- **校验位**: 无
- **流控制**: 无

#### 数据帧格式
所有通讯数据均采用统一的帧格式：

| 字段 | 长度 | 描述 |
|------|------|------|
| 帧头 | 1 byte | 固定值 0xA5 |
| 包序号 | 2 byte | 0x00-0xFF 循环递增 |
| 功能码 | 1 byte | 标识帧类型和功能 |
| 数据长度 | 1 byte | 载荷数据的字节数 |
| 数据载荷 | n bytes | 具体的命令或反馈数据 |
| 校验位 | 1 byte | 从帧头到载荷的8位累加和校验 |

#### 功能码定义

| 功能码 | 含义 | 帧类型 | 通讯方向 | 周期占用 |
|--------|------|--------|----------|----------|
| 0x4A | MIT电机控制帧 | 定频帧 | 主机→MCU | 占用下行周期 |
| 0x4D | MIT电机反馈帧 | 定频帧 | MCU→主机 | 占用上行周期 |
| 0x5A | 电机零位设置帧 | 信令帧 | 主机→MCU | 占用下行周期 |
| 0x5D | 零位设置反馈帧 | 信令应答帧 | MCU→主机 | 占用上行周期 |
| 0x6A | 定频通信频率设置帧 | 信令帧 | 主机→MCU | 占用下行周期 |
| 0x6D | 通信周期设置反馈帧 | 信令应答帧 | MCU→主机 | 占用上行周期 |
| 0x7A | 错误清除帧 | 信令帧 | 主机→MCU | 占用下行周期 |
| 0x7D | 错误清除反馈帧 | 信令应答帧 | MCU→主机 | 占用上行周期 |
| 0x8A | 查询MCU和电机丢包率 | 信令帧 | 主机→MCU | 占用下行周期 |
| 0x8D | 回复MCU和电机丢包率 | 信令应答帧 | MCU→主机 | 占用上行周期 |
| 0x9A | 外设状态设置帧 | 信令帧 | 主机→MCU | 占用下行周期 |
| 0x9D | 外设数据上报帧 | 信令应答帧 | MCU→主机 | 占用上行周期 |

#### 主要数据帧详解

##### 1. MIT电机控制帧 (0x4A)
**用途**: 主机向MCU发送实时电机控制命令，支持MIT模式控制

**单电机控制数据结构** (11 bytes per motor):
```c
typedef struct {
    uint8_t   mode;           // 电机模式: 0x00=失能, 0x0A=使能
    uint16_t  target_pos;     // 目标角度: [0~65535] 对应 ±12.5rad
    uint16_t  target_vel;     // 目标速度: [0~4096] 对应 -45~45rad/s
    uint16_t  kp;             // 位置刚度: [0~4096] 对应 0~500
    uint16_t  kd;             // 速度阻尼: [0~4096] 对应 0~5
    uint16_t  target_q;       // 目标力矩: [0~4096] 对应 -30~30N.m
} MOTOR_CON_T;
```

**完整控制帧结构**:
```c
typedef struct {
    uint8_t      head;           // 帧头: 0xA5
    uint16_t      seq_num;        // 包序号: 0x00-0xFF循环
    uint8_t      cmd;            // 功能码: 0x4A
    uint8_t      length;         // 载荷长度: MOTOR_NUM * 11
    MOTOR_CON_T  motors[MOTOR_NUM]; // 电机控制数据数组
    uint8_t      checksum;       // 校验和
} MOTOR_CONTROL_FRAME;
```

##### 2. MIT电机反馈帧 (0x4D)  
**用途**: MCU向主机上报实时电机状态反馈

**单电机反馈数据结构** (10 bytes per motor):
```c
typedef struct {
    uint8_t   work_status;    // 工作状态: 0=失能, 1=使能, 2=离线
    uint16_t  current_pos;    // 当前角度: [0~65535] 对应 ±12.5rad
    uint16_t  current_vel;    // 当前角速度: [0~4096] 对应 -45~45rad/s
    uint16_t  current_torque; // 当前力矩: [0~4096] 对应 -30~30N.m
    uint8_t   temperature;    // 当前温度 (摄氏度)
    uint8_t   error_code;     // 故障码 (见故障码表)
} MOTOR_FDB_T;
```

**故障码定义**:
- 0x08: 超压
- 0x09: 欠压  
- 0x0A: 过电流
- 0x0B: MOS过温
- 0x0C: 电机线圈过温
- 0x0D: 通讯丢失
- 0x0E: 过载

##### 3. 详细信令帧使用说明
参见上述"详细信令帧说明"章节。

#### 数据转换函数
协议采用定点数编码，需要浮点数与整数之间的转换：

```c
// 浮点数转无符号整数
int float_to_uint(float x_float, float x_min, float x_max, int bits) {
    float span = x_max - x_min;
    float offset = x_min;
    return (int)((x_float - offset) * ((float)((1 << bits) - 1)) / span);
}

// 无符号整数转浮点数  
float uint_to_float(int x_int, float x_min, float x_max, int bits) {
    float span = x_max - x_min;
    float offset = x_min;
    return ((float)x_int) * span / ((float)((1 << bits) - 1)) + offset;
}
```

**主要参数转换关系**:
- **角度**: [0~65535] ↔ [-12.5, +12.5] rad (16bit)
- **速度**: [0~4096] ↔ [-45, +45] rad/s (12bit)  
- **力矩**: [0~4096] ↔ [-30, +30] N·m (12bit)
- **刚度**: [0~4096] ↔ [0, 500] (12bit)
- **阻尼**: [0~4096] ↔ [0, 5] (12bit)

#### 校验算法
采用8位累加和校验：将帧头到数据载荷的所有字节相加，取低8位作为校验值。

#### 详细信令帧说明

**1. 电机零位设置** (0x5A/0x5D):
- 主机发送: `A5 xx 5A 01 01 checksum` 
- MCU应答: `A5 xx 5D 01 [00|01] checksum` (0x00=失败, 0x01=成功)

**2. 通信频率设置** (0x6A/0x6D):
- 主机发送: `A5 xx 6A 01 [freq] checksum` (freq单位:10Hz, 范围10-500Hz)
- MCU应答: `A5 xx 6D 01 [00|01] checksum` (0x00=失败, 0x01=成功)
- 示例：设置500Hz频率: `A5 00 6A 01 32 42` (50×10Hz=500Hz)

**3. 错误清除** (0x7A/0x7D):
- 主机发送: `A5 xx 7A 01 01 checksum`
- 功能：清除所有电机的故障状态
- MCU应答: `A5 xx 7D 01 [00|01] checksum` (0x00=清除失败, 0x01=清除成功)
- 示例：
  - 清除故障命令: `A5 00 7A 01 01 21` (假设包序号为0)
  - 清除成功应答: `A5 00 7D 01 01 24` (假设包序号为0)
  - 清除失败应答: `A5 00 7D 01 00 23` (假设包序号为0)

**4. 丢包率查询** (0x8A/0x8D):
- 主机发送: `A5 xx 8A 01 01 checksum`
- MCU应答结构：
```
帧头: 0xA5
包序号: xx
功能码: 0x8D  
数据长度: MOTOR_NUM * 2 (每个电机2字节)
数据载荷: [motor1_loss_rate] [motor2_loss_rate] ... [motorN_loss_rate]
校验和: checksum
```

**丢包率数据格式**:
- 每个电机占用2字节 (uint16_t)
- 数据范围: 0~1000，对应 0.0%~100.0%
- 单位: 0.1% (即数值1表示0.1%丢包率)
- 字节序: 高位在前，低位在后

**示例**：6个电机，丢包率均为0.1%
```
A5 00 8D 0C 00 01 00 01 00 01 00 01 00 01 00 01 44
```
解释：
- `A5`: 帧头
- `00`: 包序号  
- `8D`: 功能码(丢包率应答)
- `0C`: 数据长度(6个电机×2字节=12字节)
- `00 01`: 1号电机丢包率0.1% (数值1)
- `00 01`: 2号电机丢包率0.1% (数值1)  
- `00 01`: 3号电机丢包率0.1% (数值1)
- `00 01`: 4号电机丢包率0.1% (数值1)
- `00 01`: 5号电机丢包率0.1% (数值1)
- `00 01`: 6号电机丢包率0.1% (数值1)
- `44`: 校验和

**5. 外设状态设置** (0x9A/0x9D):

**主机发送设置命令** (0x9A):
- 功能码: 0x9A  
- 载荷长度: 1字节
- 载荷格式 (LED控制字节):

```
Bit[1:0]   红灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[3:2]   绿灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[5:4]   蓝灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[7:6]   预留 (设置为0)
```

**示例**：红灯常亮、绿灯和蓝灯熄灭
```
控制字节 = 0b00000001 = 0x01
完整帧：A5 00 9A 01 01 41 (假设包序号为0)
校验和计算：A5 + 00 + 00 + 9A + 01 + 01 = 41
```

**MCU上报外设状态** (0x9D):
- 功能码: 0x9D
- 载荷长度: 6字节

**载荷格式**:

BYTE1 (LED状态) - 与设置命令相同格式:
```
Bit[1:0]   红灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[3:2]   绿灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[5:4]   蓝灯状态: 0=熄灭, 1=常亮, 2=闪烁, 3=呼吸灯
Bit[7:6]   预留
```

BYTE2 (按键状态):
```
Bit[0]     按键1状态: 1=按下, 0=松开
Bit[1]     按键2状态: 1=按下, 0=松开
Bit[2]     按键3状态: 1=按下, 0=松开
Bit[3]     按键4状态: 1=按下, 0=松开
Bit[4]     按键5状态: 1=按下, 0=松开
Bit[5]     按键6状态: 1=按下, 0=松开
Bit[6]     按键7状态: 1=按下, 0=松开
Bit[7]     按键8状态: 1=按下, 0=松开
```

BYTE3~BYTE4 (摇杆X坐标, 大端序):
```
BYTE3      X坐标高8位
BYTE4      X坐标低8位
范围: 0~4096, 中间值 2048
  0    = X轴最小值
  2048 = X轴中间位置
  4096 = X轴最大值
```

BYTE5~BYTE6 (摇杆Y坐标, 大端序):
```
BYTE5      Y坐标高8位
BYTE6      Y坐标低8位
范围: 0~4096, 中间值 2048
  0    = Y轴最小值
  2048 = Y轴中间位置
  4096 = Y轴最大值
```

**示例**：红灯常亮、绿灯和蓝灯熄灭，第2号按键按下，摇杆位于中间位置
```
BYTE1 = 0x01 (红灯常亮)
BYTE2 = 0x02 (按键2按下)
BYTE3 = 0x08 (X坐标高8位: 2048 = 0x0800)
BYTE4 = 0x00 (X坐标低8位)
BYTE5 = 0x08 (Y坐标高8位: 2048 = 0x0800)
BYTE6 = 0x00 (Y坐标低8位)

完整帧：A5 00 9D 06 01 02 08 00 08 00 C1 (假设包序号为0)
校验和计算：A5 + 00 + 00 + 9D + 06 + 01 + 02 + 08 + 00 + 08 + 00 = C1
```

#### 通讯时序
Table Arm 机械臂采用周期性通讯机制，通讯时序规则如下：

##### 基本时序规则
- **设备上电**: 设备接通电源后，自动以默认通讯频率开始上报MIT电机反馈帧(0x4D)
- **频率配置**: 主机可通过通信频率设置帧(0x6A)修改通讯周期，设置完成后生效
- **周期性通讯**: 配置完成后，主机应以相同频率下发MIT电机控制帧(0x4A)

##### 周期内帧互斥原则
**重要**: 在一个通讯周期内，帧发送遵循互斥原则：

**主机侧（下行）**:
- 每个周期内只能发送**一种类型**的帧
- 优先级: 信令帧 > MIT电机控制帧
- 如果需要发送信令帧，则该周期不发送MIT电机控制帧

**MCU侧（上行）**:
- 每个周期内只能发送**一种类型**的帧  
- 优先级: 信令应答帧 > MIT电机反馈帧
- 如果需要发送信令应答帧，则该周期**不会**上报MIT电机反馈帧

##### 典型通讯序列

**1. 设备初始化阶段**:
```
周期1: 主机→空 / MCU→0x4D(默认频率反馈)
周期2: 主机→0x6A(设置频率) / MCU→空
周期3: 主机→空 / MCU→0x6D(频率设置应答)  
周期4: 主机→0x5A(零位设置) / MCU→空
周期5: 主机→空 / MCU→0x5D(零位设置应答)
周期6: 主机→0x7A(清除错误) / MCU→空
```

**2. 正常运行阶段**:
```
周期N: 主机→0x4A(MIT控制) / MCU→0x4D(MIT反馈)
周期N+1: 主机→0x4A(MIT控制) / MCU→0x4D(MIT反馈)
周期N+2: 主机→0x8A(查询丢包率) / MCU→空
周期N+3: 主机→空 / MCU→0x8D(丢包率应答)
周期N+4: 主机→0x4A(MIT控制) / MCU→0x4D(MIT反馈)
```

##### 信令帧处理策略
- **发送时机**: 信令帧可在任意周期插入，但会占用该周期
- **应答延迟**: MCU在下一个周期回复应答帧
- **超时处理**: 信令帧超时时间建议设置为3-5个周期时间
- **重试机制**: 超时后可重新发送，但需等待当前周期结束

##### 频率配置说明
- **默认频率**: 设备上电时的初始反馈频率(通常100Hz)
- **可设置范围**: 10Hz ~ 500Hz (通过0x6A帧设置)
- **生效时机**: 频率设置成功后立即生效
- **同步要求**: 主机控制频率应与设置的反馈频率保持一致

##### 异常情况处理
- **周期错位**: 如果主机和MCU周期不同步，可能导致通讯效率下降
- **帧丢失**: 单个周期的帧丢失不影响后续通讯
- **频率变更**: 运行时变更频率需要重新同步通讯节拍
